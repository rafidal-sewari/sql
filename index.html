Lesson 1 · SELECT & FROM

Reading data from a table

SELECT chooses which columns to show. FROM picks the table. That's it — those two words form the backbone of every query you'll ever write.

Select specific columns
SELECT name, email FROM users;
Select every column
SELECT * FROM products;
💡 * is a wildcard meaning "all columns". Great for exploring — less ideal in production code.

Lesson 2 · WHERE

Filtering rows

WHERE adds a condition. Only rows that pass it come back. Without it, SELECT returns the whole table.

Operators
-- = != > < >= <= AND OR NOT SELECT * FROM users WHERE age > 25; SELECT * FROM users WHERE age > 25 AND age < 35;
💡 AND requires both conditions true. OR requires either. Parentheses control order: (age > 25 OR age < 18) AND name != 'Bob'

Lesson 3 · ORDER BY & LIMIT

Sorting and capping results

ORDER BY sorts results. ASC is low → high (default). DESC is high → low. LIMIT caps how many rows come back.

Sort by price, highest first
SELECT name, price FROM products ORDER BY price DESC LIMIT 3;
Sort by multiple columns
SELECT * FROM products ORDER BY category ASC, price DESC;

Lesson 4 · Aggregate Functions

Crunching many rows into one

Aggregate functions collapse a set of rows into a single value. The five essentials:

The five functions
COUNT(*) -- number of rows SUM(amount) -- total of a column AVG(amount) -- average value MIN(price) -- smallest value MAX(price) -- largest value
Example — order stats
SELECT COUNT(*) AS total_orders, SUM(amount) AS revenue, AVG(amount) AS avg_order FROM orders;
💡 AS renames a column in the output — called an alias. Use it to make results readable.

Lesson 5 · GROUP BY & HAVING

Aggregating by category

GROUP BY splits rows into buckets and runs aggregate functions on each. HAVING filters those buckets — it's like WHERE but for groups.

Count products per category
SELECT category, COUNT(*) AS total FROM products GROUP BY category;
HAVING — filter groups
SELECT category, COUNT(*) AS total FROM products GROUP BY category HAVING COUNT(*) > 1;
💡 WHERE filters rows before grouping. HAVING filters the resulting groups after aggregation.

Lesson 6 · JOINs

Combining tables

JOINs link two tables on a shared column. Most of the time you want INNER JOIN (only matching rows) or LEFT JOIN (all from left, even with no match).

INNER JOIN — only matched rows
SELECT users.name, orders.amount FROM users INNER JOIN orders ON users.id = orders.user_id;
LEFT JOIN + alias shorthand
SELECT u.name, COUNT(o.id) AS order_count FROM users u LEFT JOIN orders o ON u.id = o.user_id GROUP BY u.id;
💡 LEFT JOIN keeps every row from the left table — handy when you want "users with 0 orders" to still appear.

Lesson 7 · Modifying Data

INSERT, UPDATE & DELETE

These commands change data permanently. Real databases have no undo — always add a WHERE to UPDATE and DELETE, or you'll modify every single row.

INSERT — add a row
INSERT INTO users (name, email, age) VALUES ('Sam Lee', 'sam@example.com', 29);
UPDATE — change existing rows
UPDATE users SET age = 30 WHERE id = 3;
DELETE — remove rows
DELETE FROM users WHERE id = 3;
⚠️ DELETE FROM users; with no WHERE wipes the entire table. Always check the WHERE first.
